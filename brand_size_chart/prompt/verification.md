# Brand Size Chart Verification

Verify the produced artifact against the stage schema, browser-visible source evidence, and canonical selection rules. Use the configured browser when the original source must be reopened to verify the artifact. Report conflicts explicitly and never repair the artifact inside verification.

Verification must be based on the current `Stage result JSON` and the current artifact files listed in that result. Feedback from previous attempts is not evidence. Do not repeat a previous error unless it is still true in the current `Stage result JSON` and current artifact files. When reporting an untested URL, quote the exact URL from the current canonical inventory and confirm that it is absent from both opened and rejected inventory entries. A URL already present in `opened_urls` is tested. A navigation, search, home, sitemap, FAQ, or help URL is not an untested size-chart candidate merely because it is listed in the inventory; it needs rejection or opening only as a bounded discovery surface. Do not fail `source_discovery` solely because one helper URL such as navigation, sitemap, search, home, FAQ, or help is listed in `candidate_urls` but not opened or rejected when the current result already returns evidence-backed concrete size-chart candidates and that helper URL is not itself a concrete size-chart table candidate.

For `workflow_run_prompt_apply`, `source_type_allow_list` must contain only exact source type keys from the prompt context. Natural-language phrases such as `all supported source types` are not source type keys. An unrestricted request for all source types must be represented by an empty `source_type_allow_list`. Product type values must be separated from `shared_instruction`: `product_type_request_list` owns requested product type phrases, and `shared_instruction` must not repeat those values.

For `workflow_run_prompt_apply`, `priority_country_code` must reflect the priority country requested in the workflow prompt as one ISO 3166 alpha-2 uppercase country code. The stage must not invent a default country when the workflow prompt does not choose one. The priority-country request must not remain duplicated in `shared_instruction`.

For `source_discovery`, verify source-type completeness inside the bounded source surface, not only internal consistency. A passed verification requires browser-visible evidence that the main stage checked the relevant source surface for the requested source type and wrote every concrete matching unique size-chart table candidate found inside that bounded surface to `discovered_source_list`. Verification must fail an absence result unless the canonical inventory proves that the main stage did derive localized size-chart search-term families for the source language and market and ran separate browser-visible searches for each distinct localized family; these terms are search terms, not URL templates. Inside one source type, uniqueness is by `size_group_key`; verification must not require a second `discovered_source_list` item for a duplicate or equivalent table with the same `size_group_key` when one evidence-backed candidate with that key is already returned. Requested product types are coverage targets only; verification must fail when the main stage drops concrete unique tables from an opened source page only because they are outside the requested product types. Do not require status='failed' only because requested product-type coverage is incomplete when at least one concrete candidate is evidence-backed; missing requested product types must be reported in error_list with browser-backed reasons, and verification may still pass for the accepted candidates. Source discovery is failed only when no concrete acceptable candidate remains, when `discovered_source_list` items are not evidence-backed, when the canonical inventory evidence is missing or stale, when bounded candidate URLs from the inventory were neither opened nor rejected, or when source-type boundaries are confused. Empty discovery, skipped discovery, missing or stale canonical inventory evidence, untested bounded inventory URLs, or source-type boundary confusion must fail verification with actionable feedback for the same main stage. Do not fail only because unbounded search evidence contains additional similar product URLs after the inventory already proves that representative candidates were opened or rejected for the requested product type.

For `source_discovery`, enforce the same market-selection ladder as the main stage. If priority country tables exist, `discovered_source_list` must contain only tables whose `country_code_list` contains `priority_country_code`. If no priority country table exists and global tables exist, `discovered_source_list` must contain only global tables marked with `country_code_list=["GLOBAL"]`. If neither priority country nor global tables exist, European country tables may pass only when the evidence proves the relevant official European country tables do not differ and `discovered_source_list` items are marked with `country_code_list=["EU"]`. Conflicting European country tables must fail verification with the exact conflicting country codes, URLs, and `size_group_key` values.

For `table_extraction`, verify every source row, column, unit, size label, measurement, applicability description, source URL, source title, and `size_group_key` against the saved browser-visible evidence. Missing source columns or empty charts must fail verification. When raw DOM or diagnostic evidence contains hidden or non-rendered rows, compare those rows against the current extracted chart before failing. Do not report hidden or non-rendered rows as an extraction error when the current `Stage result JSON` already omits them and the current visible rows, size labels, units, and measurements match browser-visible evidence. If you report a hidden-row or stale-row error, quote the exact current extracted row, measurement name, and value that still contains the hidden data.
