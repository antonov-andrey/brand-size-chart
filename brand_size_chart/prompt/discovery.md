# Source Discovery

Use the configured browser to find sources for the requested source type. Every source page, source asset, and source data load must happen through that browser. Other loading mechanisms are forbidden for source data. Write browser-visible evidence artifacts and return only candidates backed by those artifacts.

Run discovery as a bounded workflow. Build one canonical source-surface inventory artifact named `source_surface_inventory.json` for the requested source type before writing `discovered_source_list`. That inventory must include search queries, candidate URLs, opened URLs, accepted tables, rejected URLs, rejection reasons, source-loading errors, and evidence paths. The canonical inventory path must be updated on every attempt before handoff; attempt-only inventory names are allowed only as extra diagnostics and must not replace the canonical inventory evidence referenced by `discovered_source_list` or verification.

In the canonical inventory, `candidate_urls` must contain only concrete source candidates that may directly hold size-chart data for this source type. Search pages, navigation pages, home pages, sitemap pages, FAQ index pages, help index pages, and other helper surfaces are discovery surfaces, not candidate URLs. Record helper surfaces in opened or rejected inventory entries with evidence and reasons, but do not list them as candidate URLs unless that exact opened URL itself contains a concrete size-chart table candidate. Do not put broad search-result or category product URL inventories in `candidate_urls`; if you collect broad product URL lists for sampling or coverage, store them separately as evidence fields such as `search_result_url_list`, `category_product_url_list`, or another clearly named non-candidate list, and record the sampling or rejection rule that selected the concrete candidate URLs.

Search for size-chart data in any browser-visible form: HTML table, page text, modal, rendered widget, PDF, image, embedded asset, product image, help section, FAQ section, Q&A section, seller answer, product details, or size recommendation block. These forms are universal; source types differ only by authority, location, and applicability boundaries.

When discovering official brand sources, identify browser-visible official host variants before concluding that a source type has no tables. Host variants include global brand domains, country-code brand domains, and locale paths that browser evidence shows as brand-owned. Open the relevant official host roots and inspect their visible navigation, footer, search, help, FAQ, sitemap, and static-page links. Do not stop after one official domain variant fails when another brand-owned host variant is visible or inferable from browser-visible evidence.

For each discovered official host, infer the browser-visible language and market from the opened page itself, then search for size-chart concepts in that language and market. Use the site's own search, visible navigation, footer/help pages, sitemap pages, and browser-opened public search results when they are available. Localized terms such as Turkish `beden rehberi` and `beden tablosu` are search terms, not URL templates. Record the localized queries and the opened or rejected results in the canonical inventory. Do not conclude that an official host has no size guide until localized size-chart term searches for that host are represented in the canonical inventory.

Apply one market-selection ladder for `discovered_source_list`. First search for the `priority_country_code` from `PromptScope`. If priority country tables exist for the requested source type, `discovered_source_list` must contain only items whose `country_code_list` contains that priority country code. Record every opened lower-priority table in the canonical inventory as rejected with reason `priority_country_available`; never include it in `discovered_source_list`. If no priority country table exists, include global tables only when global tables exist and mark them with `country_code_list=["GLOBAL"]`. If neither priority country nor global tables exist, include European country tables only after verifying that the relevant official European country tables do not differ; mark verified consensus items with `country_code_list=["EU"]`. If European country tables differ, return `status="failed"` with exact conflicting country codes, URLs, and `size_group_key` values instead of selecting one country.

On retry, treat `Previous stage result JSON` as stage input. Preserve evidence-backed accepted candidates from the previous result unless verification feedback identifies a concrete error in that candidate. Complete the missing bounded inventory, rejection reasons, or coverage around those candidates instead of discarding them, then overwrite the canonical inventory artifact so verification sees the completed state.

Return every concrete unique size-chart table candidate found inside the bounded source surface that matches the source type. One page with multiple tables must produce multiple candidates when those tables have different `size_group_key` values. Do not return aggregate page-level candidates. Inside one source type, return at most one `discovered_source_list` item for one `size_group_key`; if another page, locale, or asset exposes an equivalent table for the same `size_group_key`, record that duplicate or equivalent table in inventory evidence and source notes, but do not return a second candidate with the same `size_group_key`. Requested product types are coverage targets only; they must not filter or narrow `discovered_source_list`, `accepted_tables`, or table extraction scope. When an opened source page contains concrete tables that are outside the requested product types, still return those tables and leave `product_type_hint_list` empty unless the evidence explicitly covers one requested product type.

If at least one concrete candidate is evidence-backed, return status='success' even when requested product-type coverage is incomplete. Record missing requested product types in error_list with browser-backed reasons and keep the accepted candidates in discovered_source_list.

Do not silently skip a source type. If browser access, search, page rendering, selectors, sitemap traversal, candidate URL loading, or confirmed absence of matching tables prevents returning concrete candidates, return a failed terminal result with concrete blocker details and the canonical inventory evidence so the source-type summary records that source type as failed and the workflow can continue to the next source type.
