# Brand Size Chart DBOS Workflow

## Scope

This repository owns a DBOS workflow container that parses one `brand_list`, uses one `secret` reference for external runtime credentials, and writes canonical `brand_size_chart` artifacts plus `brand_size_chart_audit` evidence.

## Determinism Boundary
Shared workflow-container authoring rules belong to the `workflow-container-developer` plugin reference `references/workflow-container-authoring.md`. Generic `Codex` subprocess execution, JSON schema output boundary, generic prompt rendering, generic prompt partials, generic browser-tool event validation, and generic JSON artifact writing belong to `workflow-container-runtime`. This document defines only `brand-size-chart` domain behavior: source types, source discovery semantics, table extraction semantics, `size_group_key`, mechanical domain guards, output paths, and local production runtime facts.

Shared workflow-container ecosystem code quality rules also belong to the `workflow-container-developer` plugin reference `references/workflow-container-authoring.md`; this document does not restate them.

Real source discovery and table extraction are Codex-owned browser stages. Python code must not parse marketplace or brand pages into size-chart data and must not load source pages or source assets through non-browser request paths. The `source_discover` and `table_extract` stages must call Codex with configured browser access and return schema-valid domain results.

## Stage Workflow Contract

The shared action and verification stage shape belongs to the `workflow-container-developer` plugin reference `references/workflow-container-authoring.md`. The canonical action stage keys for this workflow are `workflow_run_prompt_apply`, `source_discover`, `table_extract`, `coverage_decide`, and `canonical_select`.

`source_discover` must build source-surface inventory before it returns its delta result. The inventory evidence must cover discovery queries, URL entries, accepted tables, non-returned tables, and rejection reasons for the requested source type. Blocking source-access errors belong to the stage result `browsing_error_list`. Codex writes selected concrete table candidates only to `SourceSurfaceInventory.table_list` with `state="accepted"`. Python builds the cross-stage `SourceDiscoveryResult.discovered_source_list` from accepted inventory rows after mechanical validation. `SourceSurfaceInventory.url_list` is the only URL inventory list; URL state is owned by each URL row's `state` field. Inside one source type, one `size_group_key` may appear at most once among accepted table rows; duplicate, equivalent, market_conflict, market_filtered, or rejected tables for the same `size_group_key` stay in the canonical inventory with evidence-backed reasons and must not become separate accepted rows.

SourceSurfaceInventory shape is owned by the `SourceSurfaceInventory` Pydantic model. Before `source_discover` runs, the stage writes that model's generated JSON schema to `state.schema.json` in the stage directory. The prompt derives that schema path from `prompt_context.json` instead of receiving a separate schema-path field. `state.json` must conform to that schema and remains the canonical source-surface inventory artifact.

source_discover follows this sequence: initialize the canonical inventory, discover source surfaces, build `product_type_sex_worklist` for product-scoped sources, close selected source boundaries according to the source-boundary closure contract, enumerate concrete tables inside opened bounded surfaces, apply market selection before writing final table states, write selected tables to `table_list` with state `accepted`, write non-returned concrete tables to `table_list` with state `duplicate`, `equivalent`, `market_conflict`, `market_filtered`, or `rejected`, and write the delta stage result only after the inventory is complete. For product-scoped source types, `product_type_sex_worklist` is derived from the same deterministic sex-row contract as the `source_discover` prompt. Start from the closed source-discovery sex set: women, men, girls, boys, unisex, and unisex_child. If the requested product type names an explicit sex, keep only that sex. If the requested product type does not name an explicit sex, create one worklist row for every sex in the closed set. A removed product type and sex row must remain in product_type_sex_worklist with state='rejected', reason and evidence_path_list that prove the pair is impossible for this product type. Worklist rows that still need source coverage use state='active'. Each worklist row has one `worklist_key`, and each URL, accepted, or market_filtered inventory entry that covers an active row lists that key in `worklist_key_list`. The canonical inventory must contain one URL, accepted, or market_filtered inventory entry for each active worklist row unless one evidence-backed accepted table explicitly covers multiple rows and lists every covered `worklist_key`. Duplicate and equivalent table rows are audit/completeness rows only; they do not close active worklist rows, and the accepted table row they duplicate or equal must list every covered `worklist_key`. `market_conflict` rows are fatal blockers and do not close active worklist rows. Rejected worklist rows are terminal by their own `state`, `reason`, and `evidence_path_list` and do not require a second URL or table inventory entry.

For official hosts, `source_discover` must infer the browser-visible language and market from the opened page, derive localized size-chart search-term families for that language and market, and run separate browser-visible searches for each distinct localized family before concluding that no official size-guide table exists. These are search-term families, not URL templates.

`product_type_request_list` defines coverage targets and later-source-type continuation only. `official_brand_size_guide` and `official_seller_size_guide` must receive an empty product-type scope; they search all size-guide tables available on their non-product source surfaces. Product-type scope is passed only to source types that inspect concrete products. It must not filter accepted table rows or table extraction scope for an opened source surface. Concrete tables found inside the source-type boundary must be accepted even when they do not cover any requested product type.

`priority_country_code` defines the market priority for source discovery. It is parsed from the workflow-run prompt by `workflow_run_prompt_apply` like `source_type_allow_list` and `product_type_request_list`; no default country exists in code, prompts, or design. Every accepted table row must declare `country_code_list`. For each source type, accepted table rows follow one strict ladder: if priority country tables exist, accept only rows whose `country_code_list` contains `priority_country_code`; otherwise, if global tables exist, accept only rows marked with `country_code_list=["GLOBAL"]`; otherwise accept European consensus rows marked with `country_code_list=["EU"]` only when evidence proves the relevant official European country tables do not differ. Conflicting European country tables are recorded only as `SourceSurfaceInventory.table_list` rows with `state="market_conflict"` and make `source_discover` fail with exact country codes, URLs, and `size_group_key` values.

`source_discover` must not silently skip. Zero accepted table rows without evidence-backed terminal no-table reasons in `state.json`, missing inventory evidence, incomplete source loading, or conflicting European country tables is a critical incomplete stage result. The workflow retries it through the action-stage feedback loop and fails after the attempt limit instead of producing an empty successful source type. A semantically verified inventory with zero accepted table rows and concrete no-table reasons derived from `state.json` is a non-fatal no-table source result: the source-type summary state is `skipped`, and the derived reasons are stored in `warning_list`. Missing requested product types belong only to `coverage_decide.uncovered_product_type_gap_list`. Runtime exceptions, Codex failures, European country conflicts, and stage verification exhaustion remain fatal source-type failures.

`source_discover_verify` is the verification half of `source_discover`. It checks source-type completeness and source-type boundary, not only JSON shape or self-consistency. It may use the same source access capability as `source_discover` to inspect current source state, but it does not own an artifact namespace and must not save independent evidence. If evidence is missing, it returns feedback so the next `source_discover` action attempt saves the required artifacts.

`table_extract` is one batch stage per source type. It receives the complete verified `discovered_source_list`, builds an explicit execution plan from that list, and extracts each table sequentially inside one Codex stage. `source_title` is the browser-visible source identity for selecting the table from the source surface. `size_group_key` is the normalized artifact key derived from `source_title` and evidence through `Size Group Key Contract`; it is not a browser-visible selector unless the source itself visibly contains that exact key. Codex action returns one `TableExtractionDeltaBatchResult` with one extraction delta per discovered source in exact execution-plan order. Each delta owns only `evidence_path_list` and `applicability_description`; Python builds immutable source identity, `size_group_key`, and `country_code_list` from verified source discoveries and builds deterministic `chart_path` only from `TableExtractionPromptContext.execplan_item_list[].chart_write_target.artifact_path` into `TableExtractionArtifact` handles. Mechanical validation checks the delta count, chart files, evidence references, and execplan chart targets, but it does not build cross-stage artifact handles. Later stages receive only `TableExtractionArtifact` handles and read chart files from `chart_path` when chart content is needed. The source-type workflow must not enqueue one child workflow per table. A failed table inside the batch fails the stage and sends feedback to the same `table_extract` action stage.

The `table_extract` prompt must include a `## FSM` section for durable batch progress. The durable progress ledger is the chart artifact target declared by each `TableExtractionPromptContext.execplan_item_list[].chart_write_target`, not a separate `state.json` file. Item state follows `missing-or-invalid-chart -> valid-chart`. Retried extraction reads the ordered execplan and existing chart target files; it preserves existing valid chart artifacts unless verifier feedback names a concrete item, source title, or chart target as wrong. Successful handoff requires one delta and one valid chart artifact for every execplan item. If any chart target is missing or invalid, mechanical validation fails and the runtime feedback loop retries the same `table_extract` action stage.

`coverage_decide` compares requested product types directly against verified table artifact handles, emits one positive `CoveredProductType` entry for each explicitly covered requested product type, and emits structured `uncovered_product_type_gap_list` entries for requested product types without explicit coverage. Verified table context must include `source_url`, `chart_path`, `evidence_path_list`, `country_code_list`, `applicability_description`, `source_title`, and `size_group_key`. chart_path and evidence_path_list are artifact references, not product-type coverage evidence. coverage_decide must read every referenced chart_path and relevant evidence_path_list artifact before using those artifacts for coverage. Positive coverage must be backed by explicit applicability evidence from loaded chart content, loaded evidence content, `applicability_description`, or chart description; verification must fail when positive coverage depends only on path strings. Identifier-only fields such as `source_url`, `source_title`, `size_group_key`, and `country_code_list` are supporting context, not coverage evidence. `CoveredProductType.reason` explains one positive product-type coverage decision from one verified `chart_path` and must not explain product-level gaps.

`canonical_select` receives verified eligible `CanonicalSelectionCandidate` items. Python builds each candidate from one `TableExtractionArtifact`, source-type registry `source_priority`, and `applicability_status` computed from `country_code_list` plus prompt-selected `priority_country_code`. Codex does not compute `source_priority` or `applicability_status`; it uses those candidate decision values when selecting canonical tables. `canonical_select` groups candidates by `table_extraction_artifact.size_group_key`, selects the candidate with maximum `source_priority` inside each group, and returns only `selected_chart_path` for each selected canonical table. Final brand output copies the selected chart by `selected_chart_path` and derives the output `size_group_key` from the selected `TableExtractionArtifact`; it does not rebuild a second identity key from source type or source URL. Same-priority candidates may be selected as equivalent only after reading every candidate `chart_path` as a local JSON artifact and proving they are equivalent by normalized chart content and applicability scope. Normalized chart content means row `size_label` and measurement `name`, `unit`, `min_value`, and `max_value`; applicability scope means `applicability_status`, `country_code_list`, `applicability_description`, `evidence_path_list`, `source_url`, and `source_title`. source_url and source_title are compared evidence fields, not equivalence keys. Equivalent maximum-priority candidates are selected deterministically by sorting candidates by `chart_path`, then `source_url`, then `source_title` and choosing the first candidate. Unresolved same-priority equivalence omits canonical selection for that `size_group_key`; Python derives unresolved conflict candidates from omitted groups and `CanonicalSelectionCandidate` context instead of reading a Codex-owned conflict payload.

Size-chart data may appear as an HTML table, page text, modal, rendered widget, PDF, image, embedded asset, product image, help section, FAQ section, Q&A section, seller answer, product details, or size recommendation block. These forms are universal across source types; source types differ only by authority, location, and applicability boundaries. A separate official asset source type is forbidden.

`official_brand_size_guide` covers official brand-owned non-product size-guide surfaces on the brand website. `official_seller_size_guide` covers official or authorized reseller or distributor non-product size-guide surfaces for the brand when the brand sells through an official seller in a relevant country; evidence must prove the seller is official or authorized.

`official_brand_product_page`, `official_marketplace_product_page`, and `official_marketplace_store` are product-type-scoped source types. When `product_type_request_list` is empty, the workflow must not run those source types. When `product_type_request_list` is present, each source discovery receives the current uncovered product-type list through `PromptScope`. After each source type, a semantic `coverage_decide` stage determines which requested product types are explicitly covered by all verified tables accumulated so far; only still-uncovered product types are passed to later source types. The brand workflow does not run a final brand-level `coverage_decide` stage. Final brand output uses the last cumulative coverage result for product-scoped runs, an empty coverage result when product scope is empty, and a deterministic uncovered-gap result when requested product types exist but no verified table artifacts were produced.

## Artifact Layout

Canonical brand output is written to `brand_size_chart/brand/<parsed_brand_key>/manifest.json`. Size chart data is written to `brand_size_chart/brand/<parsed_brand_key>/size_chart/<size_group_key>.json`. Stage result and verification artifacts are written under `brand_size_chart_audit/brand/<parsed_brand_key>/`. Runtime evidence artifacts may be written under any run-owned artifact root declared for the current workflow.

Batch table extraction uses `brand_size_chart_audit/brand/<parsed_brand_key>/source_type/<source_type>/table_extract/result.json`, `brand_size_chart_audit/brand/<parsed_brand_key>/source_type/<source_type>/table_extract/verification.json`, and generated table files under `brand_size_chart_audit/brand/<parsed_brand_key>/source_type/<source_type>/table_extract/chart/<size_group_key>.json`.

The root run result is written to `brand_size_chart_audit/run/result.json`. The result model carries `status`, `message`, and `error_list` directly; separate terminal-result artifacts are forbidden.

## Size Group Key Contract

Real size-chart table keys use only `{sex}_{product_group_or_type}` or `{sex}_{sex_suffix}_{product_group_or_type}`. `sex` is limited to `women`, `men`, `girls`, `boys`, `unisex`, and `unisex_child`. A suffix is present only when source evidence shows a real chart-group qualifier; no default suffix exists.

Approved non-age suffix terms are `plus` and `baby`. Age-range suffixes use only the template `{min}_{max}_{month|year}` and only when source evidence names the whole chart group by an age range. Concrete age intervals are never pre-approved in the prompt or design; the numbers come only from source evidence during extraction. Current approved product group terms are `upper`, `lower`, `belts`, `shoes`, `clothing`, `dresses`, `outerwear`, `underwear`, `swimwear`, `socks`, `hosiery`, `hats`, `gloves`, and `bras`. Lower-body clothing tables such as pants, skirts, shorts, trousers, jeans, and leggings use `lower` unless source evidence exposes a distinct standalone table whose meaning is not covered by `lower`.

The key must not include `size_chart`, `chart`, `product_measurement`, `product_measurements`, product ids, source-type names, brand names, coverage buckets, or diagnostic labels. Source row size values, including row-level age labels, bra sizes, or alpha sizes, are not chart-group suffixes. Semantic verification must reject alternative names for an already approved meaning and return feedback to the same action stage.

`source_title` is the browser-visible source identity for one table and must preserve the source heading, tab, section, table title, or equivalent browser-visible label. `size_group_key` is derived from `source_title` and evidence through this contract, but the normalized key does not need to appear literally in the source.

## Mechanical Guard Contract

Semantic verification owns source correctness and table accuracy, but Python code still owns mechanical invariants that do not require source interpretation.

`PromptScope` must fail when `source_type_allow_list` contains unknown source types or `stage_instruction_list` contains unknown stage keys.

`source_discover` must fail before semantic verification when zero accepted table rows have no evidence-backed terminal no-table reasons in `state.json`, when `table_list` contains one `market_conflict` row, when one duplicate or equivalent table row has no accepted table with the same `size_group_key`, when one accepted row has empty source URL or title, when one artifact reference is missing or outside allowed run artifact roots, when `state.json` violates the `SourceSurfaceInventory` schema, when `product_type_sex_worklist` contains a duplicate `worklist_key`, when one active worklist key is never listed by URL, accepted, or market_filtered inventory evidence, when one rejected worklist row has empty `reason` or empty `evidence_path_list`, when an inventory entry references an unknown worklist key, when two accepted rows in the same source type share one `size_group_key`, or when accepted `country_code_list` values violate the `priority_country_code` market-selection ladder.

`table_extract` must fail before semantic verification when `TableExtractionDeltaBatchResult.table_extraction_delta_list` length differs from verified source discoveries, `TableExtractionPromptContext.execplan_item_list` contains duplicate `size_group_key` values or duplicate chart artifact targets, one artifact reference is missing or outside allowed run artifact roots, one `TableExtractionPromptContext.execplan_item_list[].chart_write_target` points to a missing chart artifact, the chart has no rows, one row has empty `size_label`, one row has no measurements, or one measurement has empty `name`, `unit`, `min_value`, or `max_value`.

`coverage_decide` must fail when one `CoveredProductType.chart_path` does not reference one verified table artifact, when `uncovered_product_type_gap_list` contains a product type outside the current requested product-type list, when `covered_product_type_list` contains a value outside the current requested product-type list, when one requested product type is neither covered nor uncovered, when one requested product type is both covered and uncovered, or when an uncovered product type gap has no reason.

`canonical_select` must fail when one selection points to no verified `CanonicalSelectionCandidate`, repeats one `size_group_key`, selects a lower-priority candidate while a higher-priority candidate exists for the same `size_group_key`, selects a non-deterministic representative among equivalent maximum-priority candidates, or omits one eligible `size_group_key` that does not have multiple maximum-priority candidates.

`source_type_summary` and final brand output must fail when their artifact paths point outside the result directory or to missing files.

## Production Runtime Boundary

Production runtime must not import or require subagent protocol files, tracker files, Codex exec orchestration, or agent-pool state. Static prompts are repository artifacts consumed by future DBOS steps.

Every browser-backed stage must treat cookie banners, drawers, overlays, broad locators, and transient navigation errors as normal browser-interaction conditions. The stage must close or answer visible blockers before retrying one click, refine multi-match selectors through the current browser snapshot before retrying, and retry transient navigation failures such as `ERR_NETWORK_CHANGED` through the same configured browser before marking a source unavailable.

`DBOS_SYSTEM_DATABASE_URL` must be set before startup. It may use `postgresql://`, `postgres://`, or `sqlite://`. SQLite is allowed only when configured explicitly; DBOS defaults to local SQLite system storage when no system database URL is provided, and this workflow must reject that hidden fallback.

The default local private DataSource path is `.secret` under the project root. Container runtimes, including local Docker Compose and Kubernetes pods, must mount that DataSource read-only at `/input/.secret`, copy it at container startup into pod-local writable `/runtime/.secret`, and pass only `/runtime/.secret` to workflow code. `CODEX_HOME` must point to `/runtime/.secret/codex_profile` when that directory exists. Workflow code must not write to `/input/.secret`; mutable private DataSource prefixes are written back only by the platform after terminal workflow state. `openvpn/config.json` is required for production source loading, and the named `.ovpn` file must exist. `playwright_profile/**` and `codex_profile/**` are optional. When `playwright_profile/**` is absent, runtime starts from an empty browser profile. When `codex_profile/**` is absent, Codex authentication must come from the surrounding runtime environment.

The repository provides one standalone local compose profile independent of `marketplace-automation`: `vpn` starts an `openvpn` service, starts a `playwright-mcp` browser runtime service in the OpenVPN network namespace, and starts the workflow service in the normal compose network with an explicit SQLite DBOS system database under `/runtime`. The workflow image contains the installed project package; container runs must not bind-mount the repository checkout as writable runtime code. The Playwright MCP writable artifact namespace must be `/output/.playwright-mcp/current` so browser evidence can be written inside the shared workflow output root without allowing automatic Playwright MCP page or console artifacts beside root workflow outputs. Browser runtime mutable state, including persistent profile and generated MCP config, must stay under pod-local `/runtime` and must not be placed under `/output` only because browser evidence is written there.

The workflow process receives one `browser-vpn-runtime` Playwright MCP URL before the root DBOS workflow starts. DBOS stages receive only that run-level MCP URL. The workflow process must not run in the OpenVPN network namespace; dependency installation, DBOS, Codex, and non-browser network calls must use the normal network path. Stage prompts, stage code, and stage arguments must not customize profile paths, VPN paths, MCP commands, browser flags, locale, timezone, user agent, or stealth behavior. Those settings are owned by `browser-vpn-runtime`.

Search queries inside browser-backed stages use Codex internal web search. Playwright MCP must open only target source pages selected from internal search results, site navigation, saved evidence, or prompt context. Browser-backed `source_discover` and `table_extract` results expose URL-level failures through `browsing_error_list` items with `url` and `error`.

Local standalone Docker builds receive `workflow-container-runtime` through the `WORKFLOW_CONTAINER_RUNTIME_CONTEXT` build context, defaulting to `../workflow-container-runtime`. The workflow image installs that context as a separate Python package before installing `brand-size-chart`; it must not clone `workflow-container-runtime` from GitHub during Docker build and must not copy runtime-owned code into `brand_size_chart/**`.

Domain static prompts for this project live under `brand_size_chart/prompt/template/`. Project-local prompt fragments under `brand_size_chart/prompt/template/partial/` may contain only domain-owned fragments such as `size_group_key` rules. Generic prompt fragments such as runtime source access, artifact reference, retry context, and semantic verification preambles are loaded from `workflow-container-runtime` package resources. Root-level Markdown prompt copies under `brand_size_chart/prompt/*.md` are forbidden because they create a second prompt source of truth.

Browser evidence for this workflow is written under declared browser evidence write directories inside the current run output root. Chart artifacts, `result.json`, `verification.json`, audit JSON, and local evidence files derived from returned page data are written as declared by `Artifact Layout`. Stage owners must create declared chart and browser evidence directories before launching the Codex stage.
